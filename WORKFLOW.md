# 单任务闭环规则（WORKFLOW）

## 1. 目标
保证任何任务都按统一流程推进，避免“做了一半不知道算不算完成”。

## 2. 状态定义
- **TODO**：任务定义完整，待开始
- **DOING**：正在执行，仅允许 1 个主任务处于该状态
- **VERIFY**：已完成实现，正在验收
- **DONE**：验收通过，结果已记录

## 3. 状态流转规则
1. `TODO -> DOING`
   - 进入条件：
     - 任务有明确输入/输出
     - 验收标准可执行
   - 动作：
     - 在 progress.txt 写入“开始”记录

2. `DOING -> VERIFY`
   - 进入条件：
     - 产出物已生成
     - 自检脚本/手工检查已执行
   - 动作：
     - 记录自检结果（通过/失败）

3. `VERIFY -> DONE`
   - 进入条件：
     - 验收标准逐条满足
     - Git 变更已提交
   - 动作：
     - 在 progress.txt 记录完成时间与提交号

4. `VERIFY -> DOING`（返工）
   - 触发条件：
     - 任一验收项失败
   - 动作：
     - 记录失败原因和修复计划

## 4. 单任务约束
- 同一时间只允许一个主任务处于 DOING
- 新任务启动前，必须关闭当前 DOING（DONE 或明确阻塞）
- 阻塞超过 30 分钟必须记录阻塞项和求解路径

## 5. 完成定义（DoD）
任务标记 DONE 必须同时满足：
1. 产出物存在且可读取
2. 验收标准全部通过
3. progress.txt 有结论记录
4. Git commit 已生成

## 6. 自我进化回路（每轮必做）
在任务进入 DONE 后，追加一个固定动作：
1. 记录 1-3 条改进点（流程/工具/架构）。
2. 判断是否“可复用 + 能降本/降风险”。
3. 满足条件则：
   - 更新 architecture.md
   - 在 task.json 新增或更新对应任务
4. 在 progress.txt 写明“本轮系统进化内容”。

## 7. 异常处理
- 文件冲突：先保存现场，后最小化合并
- 验收无法执行：标记阻塞并说明原因
- 范围膨胀：拆分为新任务，不在当前任务偷加范围
